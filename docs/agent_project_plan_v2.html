Tier 1 设计理念 架构决策的元原则，动手前先对齐 ▼
架构线系统设计原则
Action / Observation 对称
所有操作是强类型 Action，所有反馈是对称的 Observation。CodeAction 是 Action 的子类，不破坏类型系统。
Runtime 解耦
Agent 核心不知道自己在哪里执行。本地进程、Docker、远程沙箱对 Agent 透明，接口不变。
单 Agent 优先，多 Agent 预埋
第一个可交付产品是完整的单 Agent。多 Agent 协调层（AutoGen 风格）作为上层抽象预留接口，不污染单 Agent 核心。
Thought 旁路存储
Thought 不是 EventStream 的一等成员，通过 event_id 映射关联。写入、回放、检索能力存在，是否开启可配置。
执行线工程实施原则
可配置优先于硬编码
Thought 写入、审计 Agent 开关、沙箱后端、存储后端——凡是"不是每次都需要"的能力，都做成配置项而非开关分支。
接口先于实现
LLMClient、ThoughtStore、AuditAgent 先定义 Protocol，再写具体实现。多 Agent 层的接口在 Phase 1 末就要定义，哪怕实现是空的。
可观测性内置
结构化日志、EventStream replay、ThoughtStore 查询从一开始就是一等能力，不是事后补的调试工具。
审计不依赖 LLM 自觉
CodeAction 的 tracking 字段由 schema 强制声明，沙箱自动插桩，不依赖 LLM 在代码里主动写 logging。

Tier 3 详细规划 按周展开，每步有明确产出和验收标准 ▼
Phase 1 — 地基（Week 1–3）目标：最小可运行单 Agent
1.1Action / Observation 基类 + 类型注册P0架构
定义 BaseAction / BaseObservation dataclass，event_id / timestamp 等公共字段，ActionRegistry 自动收集子类。同步定义多 Agent 层的 OrchestratorProtocol（空实现占位）。
产出：types/验收：所有 Action 可序列化为 JSON
1.2EventStream（内存版）P0架构
实现 append / iter / replay 三个核心方法，内存存储，接口和后期 SQLite 版保持一致。ThoughtStore Protocol 同步定义，NullThoughtStore 默认实现。
产出：core/event_stream.py验收：可 replay 完整会话
1.3LLMClient 封装 + Runtime 接口P0执行
封装 Anthropic SDK，支持 streaming，统一 tool_use 消息格式。定义 Runtime Protocol（execute_action → Observation），本地进程版实现。
产出：llm/ + runtime/验收：可换模型零修改上层
1.4基础 Tool 集（文件 + Shell）P0执行
实现 ReadFileTool / WriteFileTool / ListDirTool / RunCommandTool，每个工具产出对应的 Observation 类型，路径安全检查，命令超时控制。
产出：tools/fs/ + tools/shell/验收：全部有单元测试
1.5ReAct 主循环 + CLIP0执行
实现 Thought→Action→Observation 循环，事件写入 EventStream，支持最大步数限制和 AgentFinishAction 终止。Typer CLI 入口，--model / --workspace 参数。
产出：core/loop.py + cli/验收：能完成"写一个 hello world 并运行"任务
Phase 2 — CodeAction + 审计（Week 4–6）目标：安全可审计的代码执行
2.1CodeAction schema 设计P1架构
定义 CodeAction（intent: str, code: str, language: str, tracking: TrackingConfig），TrackingConfig 声明副作用范围（文件列表、变量名），对应 CodeExecutionObservation 结构。
产出：types/code_action.py验收：schema 可做 LLM 输出的 JSON Schema
2.2沙箱执行器 + 自动插桩P1执行
subprocess 沙箱实现，根据 TrackingConfig 在代码前后自动注入文件 diff 快照和变量追踪 logging，不依赖 LLM 自觉写日志。超时、内存限制、危险命令拦截。
产出：runtime/sandbox.py验收：文件变更被完整记录
2.3审计 Agent 实现P1执行
LLM 驱动的 AuditAgent，输入：intent + code + 执行产出，输出：结构化事实（文件变更、变量值）+ verdict（符合预期 / 偏差描述）。主 Agent 消费 verdict，可回溯原始事实。
产出：agents/audit_agent.py验收：误判率 < 10%（手动评估 20 例）
2.4ThoughtStore SQLite 实现P1执行
实现 SqliteThoughtStore，ThoughtRecord 包含 event_id / phase / content / timestamp / metadata，支持按 session_id 回放和按 event_id 查询。ThoughtConfig 控制 enabled / phases / backend。
产出：memory/thought_store.py验收：关掉后零性能影响
Phase 3 — 记忆 + 持久化（Week 6–8）目标：单 Agent 完整版
3.1ConversationHistory + token 预算P1执行
实现滑动窗口截断，tiktoken 计数，保留系统 prompt + 最近 N 轮 + 当前任务关键 Observation，超出预算时按优先级剪裁而非直接截断。
产出：memory/history.py
3.2RAG 代码语义检索P1执行
tree-sitter 解析代码结构，ChromaDB 存储函数/类 embedding，Agent 推理前自动检索相关代码片段注入 context。增量索引，文件修改后自动更新。
产出：memory/rag.py
3.3EventStream 持久化 + 会话恢复P1执行
SQLite 版 EventStream，接口和内存版完全一致（Protocol 保证），支持从中断点恢复会话，历史任务可完整 replay。
产出：core/event_stream_sqlite.py验收：内存版 / SQLite 版可热切换
3.4Benchmark 评测 + 测试集P1架构
设计 10–20 个标准编码任务，自动运行并记录成功率 / token 消耗 / 步数。开启 ThoughtStore 时自动积累评估集，Thought + Event 联合记录为后续分析准备数据。
产出：scripts/benchmark.py里程碑：单 Agent 完整版交付
Phase 4 — 多 Agent 编排（Week 9–12）目标：多 Agent 版本
4.1AgentOrchestrator 实现P2架构
基于 A3 接口实现编排层，Planner Agent 生成子任务列表，Executor Agent（复用单 Agent 核心）并行或串行执行，结果汇总回主任务。单 Agent 核心零修改。
产出：orchestration/
4.2跨 Agent 可观测性P2执行
多 Agent 的 EventStream 通过 session_id / parent_session_id 形成树状关联，ThoughtStore 支持跨 Agent 回放，审计 Agent 可访问子 Agent 的完整上下文。
产出：扩展 EventStream + ThoughtStore
4.3经验积累 + 推理模式挖掘P3架构
对积累的 Thought 数据做向量化，聚类找推理偏差模式，成功任务的 Thought 链提取为 few-shot 示例，在后续任务推理前自动检索注入。
需要足量数据P3 未来
