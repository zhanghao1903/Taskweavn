## 1. 这份讨论稿要解决什么

本稿不回答“怎么实现”，而回答：

1. TaskWeavn 到底是什么产品（产品定位）
2. 它第一阶段应该解决什么，不应该解决什么（功能边界）
3. 在现有架构约束下，下一步可探索的产品可能性有哪些（探索空间）

本稿可作为后续你在其他会话发布 plan 的前置共识材料。

---

## 2. 当前项目共识（从文档提炼）

从现有文档看，TaskWeavn 不是“更花哨的聊天代码助手”，而是：

- **Task-first 协作系统**：Task 是一等对象，聊天/文件/diff/命令只是证据层。
- **会话内可持续执行系统**：支持长程、可插话、可确认、可回放，而不是一次性 prompt-response。
- **可治理的 Agent 执行系统**：通过风险评分、自主度门禁、确认动作，把“执行能力”纳入可控边界。
- **可演进到多 Agent 的基础平台**：当前通过强约束（串行、树形、无状态）确保可解释与稳定，再逐步松绑。

一句话：

> **TaskWeavn 的核心不是“生成代码”，而是“把目标转化为可执行、可审计、可协作的任务系统”。**

---

## 3. 建议的产品定位（建议定稿方向）

### 3.1 核心定位

**“面向复杂软件任务的 Task-first AI 协作台（协作执行层）”**

关键词：

- 复杂任务（不是单轮问答）
- Task-first（不是 chat-first）
- 协作执行（不是只给建议）
- 可控可审计（不是全自动黑盒）

### 3.2 目标用户（第一阶段）

优先服务三类用户：

1. **技术负责人/资深开发**：需要把模糊目标拆成可追踪任务并控制风险。
2. **独立开发者/小团队**：需要“半自动执行 + 明确确认点”，降低上下文切换。
3. **AI-native builder**：愿意尝试 Task tree、节点对话、执行回放的新交互范式。

不建议第一阶段面向：

- 纯非技术用户（成本高，解释成本更高）
- 企业大规模多租户治理场景（权限、审计、合规未完成）

### 3.3 与竞品心智区隔（对外叙事）

建议叙事从“能不能写代码”转为“能不能把复杂工作跑起来并可控”：

- Chat assistant：会回答
- IDE agent：会改文件
- **TaskWeavn：会把工作组织成任务并持续推进，且你能随时接管与审计**

## 4. 第一阶段产品边界：明确“做/不做”

下面是建议的边界表（按 2026-05 文档状态）：

## 4.1 必须做（MVP 必选）

1. **任务树生成与编辑（协作草案）**
   - 全局目标 -> Task Tree List
   - 节点级编辑、增删、顺序调整、状态可视
2. **节点级确认机制（风险驱动）**
   - 确认动作绑定具体 Task Node
   - 显示风险说明、选项、默认行为、超时行为
3. **单会话可回放执行流**
   - Session message stream + task 过滤视图
   - 任务状态变化与文件改动摘要
4. **统一发布边界（TaskPublisher）**
   - 不同来源（用户/协作 agent/流水线/API）统一入总线
5. **可靠性底座**
   - LLM 重试、日志可配置、会话级排障能力

> 这些能力共同构成“可执行 + 可控 + 可观察”的最小闭环。

## 4.2 明确不做（当前阶段）

1. **真 DAG 调度与复杂依赖图编辑**（先坚持树形）
2. **并发写工作区与复杂锁机制**（先坚持串行执行）
3. **常驻有状态 Agent/高级调度器**（先坚持无状态 + FIFO 能力匹配）
4. **跨会话/跨用户深度协作**（先把单 session 做扎实）
5. **重型企业能力**（RBAC、审批流引擎、合规报表等）

> 不是永远不做，而是“没有数据就不做投机复杂化”。

## 4.3 有条件做（探索项）

1. artifact_refs（轻量跨任务引用）
2. Agent 模板预热（不引入有状态依赖）
3. Task-aware 总结与记忆压缩
4. API 化发布入口（先保证幂等与审计）

---

## 5. 功能边界的“判定原则”（用于后续取舍）

后续任何需求评估，可用四个问题快速判定是否纳入：

1. **是否强化 Task-first 主线？**
   - 如果只强化聊天体验，但不改善任务协作闭环，优先级降低。
2. **是否改善可控性/可审计性？**
   - 能否让用户更清楚“发生了什么、为什么、可否介入”。
3. **是否破坏当前强约束？**
   - 若破坏（串行、树形、无状态、单工作区），需提供明确数据收益与迁移方案。
4. **是否影响实现与认知复杂度比？**
   - 高复杂度低收益功能暂缓。

## 6. 候选产品方向（探索可能性）

以下是基于现有架构可探索、且相对不破坏主线的方向：

### 方向 A：Task 协作台（核心盘）

- 做深任务树交互、节点消息、确认流、结果回放。
- 目标：形成“复杂任务执行 cockpit”。
- 指标：任务完成率、用户接管效率、确认响应时延。

### 方向 B：可控自动化流水线入口

- 让 pipeline/scheduler/API 都通过 TaskPublisher 进入同一治理面。
- 目标：把“自动化任务”也纳入同一可见/可审计空间。
- 指标：多来源任务一致性、失败可定位率、幂等发布成功率。

### 方向 C：风险分层协作体验

- 低风险默认快跑，高风险强提醒与确认。
- 目标：在“效率 vs 安全”间给用户可配置平衡。
- 指标：高风险误执行率、确认疲劳度、任务中断率。

### 方向 D：任务资产化（中期）

- 任务模板、任务复用、任务结果可检索。
- 目标：从“一次执行”走向“组织知识资产”。
- 指标：模板复用率、冷启动时间下降、跨项目迁移成功率。

## 7. 产品风险（讨论阶段先显式化）

1. **认知迁移成本**：用户习惯 chat-first，task-first 需要教育。
2. **可视复杂度风险**：任务树、消息、确认、文件变更同屏可能过载。
3. **价值证明周期**：相比“立刻给答案”，任务系统价值需要多轮任务才能显现。
4. **可靠性先决条件**：没有稳定日志/重试/可回放，产品体验会被一次失败击穿。

建议应对：

- 先打磨“默认路径”（目标输入 -> 任务草案 -> 执行 -> 回放）
- 限制 UI 同时暴露的信息密度，采用渐进展开
- 强化首屏“我现在应该看哪里、做什么”的引导

## 8. 讨论结论（供你后续开 plan 参考）

### 结论 1：定位应聚焦“Task-first 协作执行层”，而非通用聊天助手

这是当前文档最一致、也最具差异化的方向。

### 结论 2：近期必须围绕三件事构建闭环

- 任务草案可编辑
- 节点执行可控（确认/风险）
- 过程结果可回放

### 结论 3：严格执行“强约束优先”策略

在没有数据前，不引入 DAG、复杂并发、常驻有状态 agent。

### 结论 4：探索应先在“统一发布边界 + 风险分层体验”上做增量

这两条既有产品价值，又不明显破坏当前架构简洁性。

## 10. 基于你提出的三个方向：可行性与边界判断

你提到的三个场景非常关键：

1. 摄影师海量图片筛选
2. 老师课件制作
3. 汽车购买信息与推荐

这里给出一个“适配度 + 如何做 + 不该做什么”的讨论结论。

### 10.1 场景一：摄影师图片筛选（高潜力，建议优先探索）

**适配度判断：高。**

原因：

- 本质是“批处理任务 + 人工确认节点”的任务流，非常符合 Task-first。
- 可以天然分解为任务树：导入 -> 质量评分 -> 风格筛选 -> 人工复核 -> 导出。
- 用户最关心“可解释筛选依据”，这与 TaskWeavn 的可审计/可回放方向一致。

可探索的最小能力：

- 上传图片集并创建批处理 Task Tree
- 给出每张图片的评分维度（清晰度、构图、曝光、重复度）
- 用户在节点上调整阈值（例如只保留评分 > 0.8）
- 最终导出候选清单 + 理由摘要

边界提醒（当前不做）：

- 不做“自动美学绝对判断”承诺（主观性过强）
- 不做复杂版权判断/人脸合规流程（超出当前产品定位）

### 10.2 场景二：课件制作（中高潜力，建议做“HTML课件优先”）

**适配度判断：中高。**

原因：

- 课件制作是明确的多步骤任务，适合任务分解与节点编辑。
- 你提到“AI 更擅长做 HTML”，这非常合理：HTML 可迭代、可预览、可版本化。
- 相比直接生成 PPT 二进制，HTML 路径更符合现有代码与文件工作流。

建议的产品切入：

- 输入教学目标 -> 生成课件任务树（大纲、内容页、图示建议、练习题）
- 产出以 HTML/Markdown 为主，支持后续导出
- 每个 Task Node 支持“教学风格约束”（年级、时长、例子难度）

边界提醒（当前不做）：

- 不承诺“完全自动一键高质量课件最终稿”
- 不先做复杂教学平台集成（LMS 深度对接）

### 10.3 场景三：买车推荐（可做，但要非常克制）

**适配度判断：中等（可做轻决策助手，不做最终决策系统）。**

原因：

- 这是“信息收集 -> 条件澄清 -> 候选比较 -> 推荐解释”的典型任务流。
- 很适合 TaskWeavn 的“先问清意图，再组织执行”的能力。
- 但领域数据时效性强，且结果影响真实消费，错误成本更高。

建议做法：

- 先用任务节点收集约束：预算、用途、油/电偏好、空间、城市路况
- 输出候选车型列表 + 每个推荐项的理由、取舍和不确定项
- 明确标注“信息更新时间”和“建议用户二次核验”的步骤

边界提醒（当前不做）：

- 不做“完全自动下单/金融决策建议”
- 不把 LLM 幻觉结果包装成确定事实

### 10.4 三个方向的优先级建议（讨论版）

如果按“与当前架构贴合度 + 落地复杂度 + 价值验证速度”排序：

1. **图片筛选**（最匹配 Task 批处理和人机协作确认）
2. **HTML 课件制作**（强文件产出场景，便于验证任务树 + 迭代）
3. **买车推荐**（可做但需额外数据治理与免责声明）

### 10.5 对产品定位的补充（从“代码任务”扩展为“复杂数字任务”）

基于上述场景，定位可从“复杂软件任务”轻微外扩为：

> **TaskWeavn 是面向复杂数字任务的 Task-first AI 协作执行层。**

其中“复杂数字任务”包括但不限于代码、内容制作、信息筛选与对比决策。
前提是不破坏当前强约束：任务树、可确认、可回放、可审计。


### 10.6 关键共识补充：Task-first 抽象是通用层，能力差异来自工具挂载

你这次的判断非常重要：

> **只要工具可用，Task-first 架构本身就能承载这些场景。**

可把它表述为一个明确产品原则：

- Task-first 负责“任务编排、状态流转、确认治理、执行回放”
- Tooling 负责“领域动作能力”（图像分析、课件生成、信息抓取与比对）
- 新场景扩展优先走“挂载新工具 + 复用既有任务抽象”，而不是重做核心架构

这意味着：

1. 图片筛选、课件制作、买车推荐，不是三套产品，而是同一执行底座上的三个 tool-pack。
2. 产品迭代优先级可从“重做交互”转为“设计高质量工具接口和结果结构化格式”。
3. 任何新方向先验证工具链可用性，再决定是否需要新增领域专属 UI。

对应到边界上，可新增一条“默认策略”：

- **默认不改核心抽象，先通过工具扩展能力；只有当多个场景持续暴露同一结构性缺口时，才升级架构层。**

### 10.7 新暴露的系统级问题：工具供给与选择负载（比“工具可扩展”更关键）

在三个场景讨论之后，出现了一个更底层的共识：

- 工具虽然可扩展，但如果工具池持续增大，协作者（尤其是 LLM 协作者）对“系统到底能做什么”的判断会快速劣化。
- 任务拓扑质量与最终结果高度相关；一旦工具选择错误，错误会在多步任务中放大。

可归纳为：

> **Task 结果上限 = 任务拓扑质量；任务拓扑质量上限 = 能力可见性与选择质量。**

#### 10.7.1 这不是文档问题，而是结构性认知负载问题

即使工具描述准确，也会出现错选：

1. 候选空间过大（召回噪声增大）
2. 语义近邻工具冲突（“看起来都能用”）
3. 长链路误差放大（前一步小错，后续放大）
4. LLM 当轮注意力/上下文预算有限（无法稳定比较全量工具池）

因此“工具越多越好”在规划质量层面并不成立。

#### 10.7.2 GAP 定性

这个问题的主因不是 Architecture GAP，而是：

- **Capability Modeling GAP**：缺少“机器可理解的能力层”
- **Planning Governance GAP**：缺少“任务拓扑生成前后的质量门禁”

结论：核心 Task-first 架构可以保留，优先补“能力表示层 + 规划治理层”。

### 10.8 建议补齐的中间层：Tool -> Capability -> Task Node

为避免协作者直接在全量工具池里“盲选”，建议建立能力中间层：

- Tool：实现细节（provider、版本、执行约束）
- Capability：语义能力（可规划对象）
- Task Node：业务任务节点（可编辑/可审计对象）

每个 Capability 至少应包含：

- `name`
- `input_schema` / `output_schema`
- `preconditions`
- `cost_level` / `latency_level`
- `risk_level`
- `reliability_score`
- `applicable_domains`
- `anti_patterns`

这样协作者做的是“选能力”，不是“猜实现工具”。

### 10.9 拓扑质量门禁（Topology Quality Gate）v0.1

在发布/执行前引入质量检查（可先打分、后逐步升级为阻断）：

1. **可执行性**：每个节点是否可绑定至少一个 capability
2. **前置条件闭合性**：节点输入是否由上游覆盖
3. **依赖合理性**：是否存在明显先后错误
4. **验证节点完整性**：关键路径是否包含 QA/人工复核
5. **风险覆盖**：高风险节点是否具备确认策略
6. **复杂度预算**：节点数/深度是否超阈值

建议输出 `topology_score`，低分任务先回修，再执行。

### 10.10 工具暴露策略：默认不暴露全量工具池

为应对 LLM 注意力限制，建议引入“三层防错机制”：

1. **预路由（Tool Gating）**：先按任务意图与输入类型收缩到 Top-K
2. **能力优先规划（Capability-first）**：先选 capability，再映射具体工具
3. **执行前体检（Preflight Validation）**：schema、前置条件、风险/成本、替代工具检查

产品原则：

> **每个节点只暴露“当前上下文最小可用能力集”，而非全量工具。**

### 10.11 谁来提供工具：从“功能产品”走向“能力平台”

另一个关键结论是“工具供给机制”必须可扩展：

- 如果只有核心开发者硬编码提供工具，长尾需求必然失配，供给压力不可持续。
- 对单个用户极高价值、对平台低通用的场景，需要用户可自助扩展。

GAP 定性：

- **Platform GAP（一级）**
- **Governance GAP（必须同步补）**

建议的三层供给模型：

1. **官方工具层**：高频、稳定、强维护
2. **组织工具层**：团队内部连接器与流程能力
3. **用户自建层**：长尾场景自助工具/agent

### 10.12 用户自建工具/Agent 的开放边界（讨论版）

建议分级开放：

- **A. No-code/Low-code Builder（优先）**：schema + API connector + 流程编排
- **B. Script Tool（中级）**：Python/JS，强沙箱、配额、审计
- **C. Agent Template（高级）**：声明能力、依赖、风险策略后注册

为了可控开放，平台至少补齐：

1. Tool Spec 标准
2. Tool Registry（发现、版本、权限）
3. Sandbox/Policy（网络、secret、资源配额）
4. Validation Harness（上架前最小测试）
5. Observability（成功率、时延、错误分类）
6. Trust/Ranking（质量评分用于路由排序）

## 11. 将本轮讨论沉淀为可复用评估模板（不含错误示例）

为避免后续每次讨论口径漂移，建议固定使用以下五项输出：

1. **当前系统可行性**（高/中/低）
2. **能力 GAP**（缺什么）
3. **GAP 性质**（Tool/Data/UX/Governance/Platform/Architecture）
4. **场景价值**（用户价值、频率、可验证性）
5. **对系统影响**（架构影响、治理影响、实施成本）

### 11.1 GAP 性质定义（统一语义）

- **Tool GAP**：缺具体执行工具
- **Data GAP**：缺高质量/高时效数据来源
- **UX GAP**：能力可做但交互不可用
- **Governance GAP**：缺风险控制、审计、免责声明、回滚策略
- **Platform GAP**：工具供给机制不可扩展（仅中心化硬编码）
- **Architecture GAP**：核心抽象无法承载，必须改架构

### 11.2 三个场景的标准化结论（汇总版）

| 场景 | 可行性 | 主要 GAP | GAP 性质 | 场景价值 | 对系统影响 | 优先级 |
|---|---|---|---|---|---|---|
| 图片筛选 | 高 | 质量评分、相似聚类、批量导出 | Tool + 少量 UX | 高 | 中低（可控） | 1 |
| HTML 课件 | 中高 | 模板生成、结构化约束、预览迭代 | Tool + UX | 中高 | 中（可控） | 2 |
| 买车推荐 | 中 | 数据源接入、时效标注、可信比对 | Data + Governance + Tool | 中 | 中高（风险敏感） | 3 |

### 11.3 本轮讨论最终共识（用于下次 plan 前置）

1. **三类场景均在当前 Task-first 能力圈内**，短期优先“加工具/加治理”，不先改核心架构。
2. **真正系统级 GAP 是工具供给与选择机制**，不是“再多写几个工具描述”。
3. **默认不暴露全量工具池**，而是对每个任务节点暴露最小可用能力集（Top-K）。
4. **先建设 Capability 层与拓扑质量门禁**，再扩工具数量。
5. **中期必须平台化工具供给**（官方/组织/用户三层），否则长尾需求不可持续。

### 11.4 执行策略（讨论版）

- **Phase A（近期）**：Capability Registry + Topology Gate + Preflight Validation
- **Phase B（中期）**：场景工具包（image/courseware/auto-research）+ 工具质量评分
- **Phase C（后续）**：用户自建工具/Agent（分级开放 + 强治理）

> 备注：以上为讨论结论沉淀，不等同于实施计划；具体排期与验收指标在后续 plan 会话确定。

## 12. 工具层抽象升级提案：工具/Agent 文件化协议（讨论稿）

你提出的新抽象可以归纳为：

> **工具是可执行文件（Executable Artifact）；Agent 是配置文件（Config Artifact）；二者都必须遵守系统协议。**

该提案的价值在于把“供给扩展性”从人工开发流程迁移到“协议 + 文件 + 注册/发现/治理”体系，使用户可在复杂场景中自助探索并沉淀成功路径。

### 12.1 设计目标

1. **动态扩展**：支持工具动态注册、加载、升级、下线。
2. **统一约束**：通过协议保障可发现、可执行、可审计、可回滚。
3. **用户自助**：允许用户创建工具文件或由工作流生成工具文件。
4. **能力固化**：用户一次成功后，可沉淀为复用资产，追求 `success once -> success many times`。

### 12.2 抽象模型

#### 12.2.1 Tool Artifact（可执行文件）

- 本体：一个可执行文件（或可执行包）
- 要求：必须符合 Tool Protocol（接口、元数据、输入输出契约）
- 职责：执行领域动作（计算、抓取、转换、调用外部系统等）

#### 12.2.2 Agent Artifact（配置文件）

- 本体：一个配置文件（声明式）
- 要求：必须符合 Agent Protocol（能力声明、策略约束、路由规则、风险策略）
- 职责：编排能力选择与执行策略，不直接承担底层动作实现

> 关系：Agent 通过协议引用 Tool；Tool 执行动作并回传结构化结果。

### 12.3 协议优先：系统最小协议面

建议先定义最小可落地协议（v0）：

**Tool Protocol（示意字段）**

- `tool_id`, `name`, `version`
- `entrypoint`（可执行入口）
- `input_schema`, `output_schema`
- `preconditions`, `side_effects`
- `risk_level`, `timeout_ms`, `resource_limits`
- `permissions`（网络/文件/secret 范围）
- `healthcheck`（可用性探针）

**Agent Protocol（示意字段）**

- `agent_id`, `name`, `version`
- `capability_scope`
- `tool_bindings`（可调用工具集合）
- `routing_policy`（场景到能力/工具的映射规则）
- `confirmation_policy`（高风险节点确认策略）
- `fallback_policy`（失败回退与重试）

### 12.4 生命周期：文件从哪里来、放哪里、怎么被发现

你提出的“文件如何来、如何存放、如何发现”是系统职责，建议拆成三段：

1. **来源（Source）**
   - 用户上传/创建工具文件
   - 系统工作流生成工具文件（用户输入 -> 模板/生成器 -> 工具）
   - 官方/组织发布的受信工具文件

2. **存放（Storage）**
   - 统一 Artifact Store（版本化、签名、权限、可追溯）
   - 元数据索引（按场景、能力、风险、质量评分）

3. **发现（Discovery）**
   - Registry 注册 + 审核状态
   - 按任务上下文 Top-K 暴露（非全量暴露）
   - 支持能力检索与策略过滤（权限、风险、可靠性阈值）

### 12.5 用户自助探索到稳定复用：能力固化链路

建议把“探索 -> 固化”的路径产品化：

1. 用户在复杂场景中尝试（可能借助官方工具）
2. 形成一次可验证成功执行（success once）
3. 一键固化为可复用资产：
   - 工具文件（或工具配置）
   - Agent 配置
   - 任务模板（含确认/回退策略）
4. 在后续同类任务中复用，实现 success many times

这条链路本质上是把“个人经验”转成“可执行资产”。

### 12.6 与现有结论的兼容关系

该提案与前文 10.7~10.12 完全兼容，并可作为其落地形式：

- Capability 层继续保留，作为 Tool/Agent 文件化体系的语义层。
- Topology Gate 继续保留，作为执行前质量门禁。
- 工具平台化继续保留，但供给单位由“代码实现”升级为“协议文件 + 审核治理”。

### 12.7 分阶段落地建议（讨论版）

- **Phase A**：定义 Tool/Agent 协议 v0 + Registry + 基础验证器（schema/权限/healthcheck）
- **Phase B**：开放用户上传与工作流生成工具文件 + 沙箱执行 + 质量评分
- **Phase C**：支持一键固化（任务成功 -> 工具/Agent/模板资产化）+ 组织级共享市场

> 该提案仍属于讨论结论，不替代具体实施计划；详细技术方案与安全边界在后续 plan 会话展开。

## 13. LLM 能力边界与工具能力边界：引入“成本感知”的决策层（讨论补充）

你这次补充的判断非常关键：

- 很多任务（例如文件整理、结构化归档、文本归并）LLM 本身就能高质量完成。
- 实践上，很多“是否需要工具”仍然依赖 LLM 的当下判断。
- 当前主流 LLM 判断能力整体可用，但调用成本不低，必须把“成本”纳入一等约束。

### 13.1 新共识：不是“LLM or Tool”，而是“LLM-first + Cost-aware Routing”

建议把策略从二选一升级为三段式：

1. **LLM 直做优先（低风险、低副作用、纯认知任务）**
2. **LLM 规划 + Tool 执行（需要外部动作、强确定性、可审计）**
3. **Tool-only（合规/稳定性要求高，必须走确定性流程）**

本质上，LLM 与工具不是对立关系，而是“判断层与执行层”的协作关系。

### 13.2 任务分流判定：何时只用 LLM，何时必须上工具

可采用一个简化判定矩阵（v0）：

- **只用 LLM（默认优先）**
  - 目标是内容组织/归纳/改写/分类
  - 不要求调用外部系统或设备能力
  - 错误成本可控，人工复核成本低

- **LLM + Tool**
  - 需要查询外部数据、调用 API、操作文件系统/环境
  - 需要结构化输出与稳定复现
  - 需要在关键节点保留审计证据

- **Tool-only**
  - 高风险动作（删除、支付、生产配置变更等）
  - 合规强约束场景
  - 必须可确定回滚与权限隔离

### 13.3 把“调用成本”写进系统目标函数

当前如果只优化“成功率”，容易导致过度调用大模型。建议引入成本目标：

> **优化目标 = 成功率 × 质量 - 时延成本 - Token/推理成本 - 失败重试成本**

可观测指标建议：

1. 每任务 token 成本
2. 每成功任务平均成本（Cost per Success）
3. LLM-only 命中率（该用 LLM 直做的比例）
4. 不必要工具调用率（本可 LLM 直做却走了工具）
5. 成本超预算触发率

### 13.4 与现有架构的衔接：增加“策略路由器”即可

不必重构核心架构，可在现有链路增加一个轻量策略层：

- 输入：任务类型、风险等级、是否副作用、预算上限、历史成功率
- 输出：执行模式（LLM-only / LLM+Tool / Tool-only）
- 约束：预算守卫（Budget Guard）+ 风险守卫（Risk Guard）

这层可与前文 Capability Registry、Topology Gate、Preflight Validation 直接协同。

### 13.5 面向产品体验的表达建议

为避免用户误解系统“只会堆工具”，可对外明确：

- **简单任务：让 LLM 直接高效完成**
- **复杂任务：让 LLM 负责规划，让工具负责稳定执行**
- **高风险任务：让系统强制走安全工具链**

这样既承认 LLM 的判断力，也正视调用成本与稳定性边界。

### 13.6 结论（讨论版）

1. LLM 判断能力已足够承担大量轻量任务，尤其是整理/归纳类工作。
2. 工具价值主要体现在外部动作、确定性执行、审计与治理。
3. 下一步关键不是“增加工具数量”，而是“提升路由质量 + 成本治理能力”。
4. 建议把“成本感知路由”纳入后续 plan 的明确工作项。

## 14. 平台化潜力补充：从“软件能力”走向“工作流与工具市场”（远期）

你这次的判断可以沉淀为一个重要结论：

> **当前系统更像平台，而不只是单一软件。**

核心原因是：在 Task-first 抽象下，一个可重复成功的业务流程，可以被固化为系统内可复用工作流资产，而不只是一次性执行结果。

### 14.1 从单次执行到可复用产品单元

以“图片筛选”为例：

- 如果某一类用户（如摄影从业者）形成高频需求，
- 且我们在该场景获得稳定成功路径（任务拓扑 + 工具链 + 确认策略），
- 则可将其固化为标准工作流（Workflow Asset），供同类用户直接复用。

这意味着“一个 app 功能”可以在系统中表现为：

- 标准 Task Topology 模板
- 对应 Tool Pack
- 预置风险/确认策略
- 可选 Agent 配置

### 14.2 平台化的资产单位（建议）

建议把可交易/可共享对象定义为四类资产：

1. **Workflow Asset**：面向场景的一体化任务流模板
2. **Tool Asset**：可执行工具文件（协议约束）
3. **Agent Asset**：策略/路由配置文件（协议约束）
4. **Knowledge Asset**：经验参数、评分规则、最佳实践说明

当这四类资产具备版本、评分、权限、可观测性后，就具备“市场化”基础。

### 14.3 远期方向：Workflow & Tool Marketplace（非近期目标）

可以作为远期愿景（明确非当前阶段交付）：

- **Workflow Market**：按行业/任务场景分发高质量工作流
- **Tool Market**：分发可插拔工具与连接器
- **质量信号层**：成功率、复用率、评分、风险等级
- **治理层**：签名、审核、权限、分级开放、违规下架

> 该方向是长期潜力，不应反向驱动近期复杂化；近期仍以核心闭环与治理能力为先。

### 14.4 对当前阶段的现实启发

即便市场化很远，当前也可先做三件“不会浪费”的事：

1. **资产化接口先行**：Workflow/Tool/Agent 都用统一协议与版本语义
2. **复用链路先行**：支持“一次成功 -> 模板固化 -> 多次复用”
3. **质量度量先行**：从现在开始采集成功率、成本、失败模式，作为未来市场排序基础

### 14.5 结论（讨论版）

1. 当前架构潜力不止于“辅助开发工具”，而是具备平台化扩展能力。
2. 许多行业场景都可以通过“任务拓扑 + 工具链”沉淀为可复用工作流。
3. Workflow/Tool 市场可以作为远期方向，但不应抢占近期核心能力建设。
4. 近期重点仍是：把成功路径稳定复现，并逐步完成资产化与治理底座。

## 15. 关于 Memory：默认“无记忆”与研究型任务上下文治理（讨论补充）

你提出的判断非常准确：

- 当前系统默认没有长期记忆能力。
- Task 总线的任务拆分会天然降低上下文压力。
- 但研究型任务即使拆分后，单个任务仍可能出现超长上下文。
- 如果 LLM 在该任务上已经稳定可做，就不必过度干预；只有“做不好”时才引入上下文治理机制。

### 15.1 新原则：Memory 不是默认前提，而是条件性能力

建议明确为：

> **默认无长期记忆；仅在研究型长上下文任务出现质量退化时，启用分层上下文管理。**

这可以避免过早引入复杂记忆系统，也与当前“强约束优先”一致。

### 15.2 任务拆分的收益与边界

#### 收益（已成立）

1. 任务分段减少单轮上下文体积
2. 上下文作用域更清晰（按 Task Node 聚焦）
3. 错误定位更容易（可映射到节点）

#### 边界（研究型任务会触发）

1. 单节点需要汇总大量材料（论文、日志、报告）
2. 结论依赖跨节点长期线索
3. 多轮推理后信息衰减/漂移

### 15.3 建议策略：三档上下文治理（按需升级）

- **Level 0：无记忆（默认）**
  - 仅使用当前节点输入与必要上游产物
  - 适用：普通执行与短链任务

- **Level 1：任务内记忆（Session-scoped）**
  - 为当前任务维护结构化摘要与证据索引
  - 不跨会话沉淀，不做全局记忆
  - 适用：中等长度研究与分析任务

- **Level 2：研究型增强记忆（Task-scoped Retrieval）**
  - 引入可检索的“任务记忆包”（摘要、证据、引用、冲突点）
  - 每轮只注入与当前子问题最相关片段（Top-K）
  - 适用：长周期研究、文献/情报密集任务

> 升级触发条件应来自质量信号，而非默认开启。

### 15.4 触发条件：何时从“不过度干预”切到“需要治理”

建议用可观测信号触发：

1. 关键事实遗漏率上升
2. 前后结论冲突率上升
3. 引用证据不可追溯比例上升
4. 单任务重试次数异常增高
5. token 成本持续升高但质量不升反降

当满足阈值时，自动从 Level 0 升级到 Level 1/2。

### 15.5 与现有架构兼容性

该补充不要求改变核心 Task-first 架构：

- 任务树仍是主抽象
- Memory 作为可选治理层，按任务类型与质量信号启用
- 可与前文的 Topology Gate、Cost-aware Routing、Capability Registry 联动

### 15.6 结论（讨论版）

1. 当前“默认无记忆”是合理选择，不必因为概念先进而提前复杂化。
2. Task 拆分已经天然缓解上下文压力，但研究型任务仍会触达上限。
3. 对研究型任务应采用“按质量触发的分层记忆治理”，而非全局常开记忆。
4. 后续 plan 可将其定义为：**质量退化时的补救能力**，不是基础前提能力。

## 16. 优先级修正：尽早做 UI 交互原型，用真实 User Case 验证假设

你这条反馈非常关键：

> 目前不少判断仍是推测，优先应前移到“尽早做 UI 原型 + 真实 user case 验证”。

这意味着当前阶段的首要目标不应继续扩写抽象，而应快速验证“协作体验是否真的可用”。

### 16.1 为什么要前置 UI 原型验证

1. **当前结论中存在假设密集区**：包括工具暴露策略、确认负担、任务拓扑可用性。
2. **交互问题很难靠文档推导**：只有真实操作路径才能暴露断点。
3. **用户价值需要可感知演示**：原型比抽象概念更能验证是否“真有用”。
4. **可更早收敛边界**：避免在未验证前过度设计治理与平台机制。

### 16.2 建议的近期优先级（重排）

将优先级调整为：

1. **P0：UI 交互原型（可点击/可走通）**
2. **P1：3 个高价值 user case 的端到端验证**
3. **P2：基于验证结果回写抽象层设计（Capability/Gate/Protocol）**

即：先验证体验，再固化抽象。

### 16.3 原型验证最小闭环（MVP for Discovery）

原型阶段只验证四件事：

1. 目标输入后是否能快速生成“可编辑任务拓扑”
2. 协作者是否能理解并操作关键确认节点
3. 执行回放是否足够支持“为什么这样做”的解释
4. 用户是否能在一次流程中得到可感知价值

### 16.4 User Case 驱动验证清单（建议）

建议优先跑三类 case（与前文一致）：

- 图片筛选
- HTML 课件制作
- 买车推荐（轻决策辅助）

每个 case 至少采集：

1. 首次完成率
2. 完成时长
3. 人工接管次数
4. 误确认/误执行次数
5. 用户主观可用性评分

### 16.5 与当前文档其他提案的关系

本节不是否定前文，而是给出执行顺序：

- `Capability`、`Topology Gate`、`Tool/Agent Protocol` 继续保留为候选框架；
- 但其优先级应服从 UI 原型实测结果；
- 先用用户行为数据决定哪些抽象“真的值得做”。

### 16.6 结论（讨论版）

1. 近期最优先事项是 UI 交互原型，而不是继续增加推导型设计。
2. 通过真实 user case 进行端到端验证，能更快识别高价值与伪需求。
3. 架构与治理抽象应以后验验证为依据，避免前置过度设计。
