多 Agent 协作架构设计

版本 v1.0 · 2026-05-07 · TaskWeavn Phase 3–4

1设计核心理念

本架构围绕两个正交的核心原则展开，它们共同决定了多 agent 与用户协作的方式。所有架构决策都以这两个原则为出发点。

原则一

消息流替代打断

传统阻塞式 interrupt 由系统主导节奏，用户被迫响应。新模型将其替换为消息流：agent 向流发送请求，用户可回应也可不回应，等待行为由自主度配置决定。用户始终是主动方。

原则二

约束驱动的用户编排

不强迫用户理解图结构。LLM 作为编排设计者，在约束边界内生成合法的 agent 协作图。用户通过自然语言描述意图，通过 UI 进行局部微调。约束本身随系统成熟度演进。

核心模型转变

旧模型：interrupt → pause → resume   （阻塞式，系统主导）
新模型：agent 发消息 → 用户可回，也可以不回  （流式，用户主导）

旧模型：用户填写编排配置               （填空题）
新模型：对话描述意图 → LLM 生成草稿    （选择题）

旧模型：约束是事后校验                  （生成后被拒绝）
新模型：约束是生成时上下文              （天然合法输出）

2消息流模型

Agent 执行过程中所有与用户相关的通信，统一流向一条 Message Stream。用户可以实时观看、随时介入，但从不被强制阻塞。

消息流示意

INFO Planner：已生成执行计划，共 5 步，预计影响 3 个文件
ASK Executor：即将删除文件 config.py，此操作不可逆，确认继续？
等待 120s，超时将使用默认值：跳过此步骤
INFO Executor：已修改 auth.py 第 42–58 行，变更已记录
WARN Auditor：发现潜在 SQL 注入漏洞（置信度 0.87），建议人工复查

两种消息类型

类型含义Agent 行为用户行为
Informational 告知 agent 做了什么 发送后立即继续执行 可忽略，可事后查阅
Actionable 请求用户输入或确认 根据自主度决定等待策略 回应或不回应，超时行为可配置
"完全不打断"状态

当用户将自主度设为最高时，所有 Actionable 消息都配置了超时自动 proceed，消息流退化为只读执行日志。用户随时可以查看，但没有任何消息阻塞 agent 执行。这是用户的主动选择，系统只负责如实呈现任务质量与自主度的关系。

3自主度系统

自主度不是一个魔法数字，而是对"agent 遇到不确定时如何行动"的精确描述。核心是两个正交维度：触发维度（何时发送 Actionable 消息）和等待维度（发出后等多久、等不到时怎么办）。

AutonomyBehavior 配置

@dataclass
class AutonomyBehavior:
    # 触发维度：何时向消息流发送 Actionable 消息
    trigger: Literal[
        "never",             # 从不，所有消息都是 Informational
        "on_risk",           # 仅高风险操作（文件删除、执行命令等）
        "on_uncertainty",    # LLM 置信度低于阈值时
        "always",            # 每个关键操作都请求确认
    ]
    confidence_threshold: float   # 0.0~1.0，仅 on_uncertainty 生效

    # 等待维度：Actionable 消息发出后的等待策略
    wait_timeout: float | None    # 秒，None = 无限等待
    timeout_action: Literal[
        "wait",               # 继续等待（低自主度默认）
        "proceed_default",    # 超时后用最保守选择继续
        "proceed_confident",  # 超时后用 LLM 最高置信选择继续
        "skip",               # 跳过该动作
    ]
    notify_on_proceed: bool       # 超时自行决定后是否告知用户

预设自主度档位

档位triggerwait_timeouttimeout_action适合场景
全自主never——批处理、可回滚任务
风险确认on_risk300sproceed_default日常使用默认
协作模式on_uncertaintyNonewait复杂、高影响任务
全确认alwaysNonewait学习、审计场景

质量与自主度的 Tradeoff

自主度高

  • 执行不被打断，速度快
  • 用户认知负担低
  • 歧义场景 agent 自行猜测
  • 错误可能在用户不知情时累积

自主度低

  • 关键决策用户全程参与
  • 错误率低，可控性强
  • 需要用户持续关注消息流
  • 任务速度依赖用户响应时间

选择权完全在用户，系统不做道德评判。UI 应清晰呈现这一权衡，而不是隐藏它。

4对话式编排设计

用户不需要理解图结构。Orchestration Designer（一个 meta-agent）接收用户自然语言意图，在约束边界内生成合法的 agent 协作图，用户只需选择和微调。

用户自然语言意图
        ↓
OrchestrationDesigner (meta-agent)
  输入：用户意图 + ConstraintProfile + ToolRegistry
  输出：OrchestrationDraft（合法图结构 + 节点配置 + rationale）
        ↓
UI 渲染图结构  ←→  用户局部微调（选择题，不是填空题）
关键原则

约束不是事后校验，而是生成时的上下文。LLM 在约束边界内生成，输出天然合法，不会出现"UI 允许但后端拒绝"的不一致。

生成的三个阶段

# 阶段一：意图解析
用户: "我想要一个能审计代码安全漏洞并自动修复的系统"
→ 核心能力: [代码读取, 漏洞分析, 修复执行, 验证回环]
→ 风险识别: 修复执行 = 高风险，需要用户确认
→ 约束映射: 需要 auditor 节点，修复前 interrupt

# 阶段二：约束内图生成
LLM 接收 ConstraintProfile 作为 prompt 上下文
在合法拓扑范围内生成图结构，无需后端 validator 二次校验

# 阶段三：能力分配
每个 agent 节点从 ToolRegistry 中选择工具子集
工具是选择，不是生成  ← 系统安全底线

对话即 Diff，不是全量重生成

用户对草稿的反馈触发局部修改，而不是重新生成整张图。UI 高亮变更节点，用户看到的是 diff，不需要每次重新理解整张图。

用户: "让 executor 更自主一点"

DraftPatch:
  target_node: "executor"
  changes:
    autonomy_behavior.trigger: "on_uncertainty" → "on_risk"
    autonomy_behavior.confidence_threshold: 0.7 → 0.4
  reason: "降低打断频率，遇到不确定时优先尝试而非询问"

OrchestrationDraft 结构

@dataclass
class OrchestrationDraft:
    nodes: list[AgentNodeDraft]
    edges: list[EdgeDraft]
    rationale: str                     # LLM 解释设计理由，展示给用户

@dataclass
class AgentNodeDraft:
    id: str
    agent_type: AgentType
    display_name: str
    description: str                   # 此节点在本编排中的职责
    tool_set: list[ToolRef]            # 从 ToolRegistry 选出的子集
    autonomy_behavior: AutonomyBehavior
    suggested_alternatives: list[AgentType]  # 用户可替换的备选类型

suggested_alternatives 让节点替换成为选择题，用户不需要知道有哪些 agent 类型。

5约束驱动的 UI 编排

分层配置模型

Layer 3: 自定义 DAG（高级用户）
  用户在约束范围内自由连接 agent 节点，调整任意参数

Layer 2: Preset + 参数调节（中级用户）
  选择最佳实践模板，通过旋钮调整关键参数

Layer 1: Preset 直选（普通用户）
  Auto-pilot / Co-pilot / Manual / Audit-Focus

三层统一 normalize 到同一个 OrchestrationConfig，系统内部无差别处理

ConstraintProfile：约束是一等公民

@dataclass
class ConstraintProfile:
    version: str

    # 节点维度：用户能放哪些 agent
    allowed_agent_types: set[AgentType]

    # 连接维度：允许哪些边
    allowed_edges: list[EdgeRule]          # (src_type, dst_type, condition)
    forbidden_patterns: list[Pattern]      # 如：禁止 executor→executor 直连

    # 组合维度：整体约束
    required_nodes: set[AgentType]         # 如：auditor 永远必须存在
    max_parallel_branches: int

    # 元信息：每条约束的存在理由
    rationale: dict[str, str]              # constraint_key → 失败模式描述
rationale 不是注释

rationale 是松绑决策的输入——你知道移除某条约束是在赌什么失败模式。有了它，"是否松绑"变成有依据的工程决策，而不是产品直觉。

约束在 UI 中的体现

约束不是报错墙，而是自然的边界感：

约束维度UI 体现体验
节点维度palette 只显示允许的节点类型被禁止的根本不存在，不是灰掉
连接维度拖拽时实时校验，非法连接弹开操作过程中感知，不是提交时报错
参数维度面板只渲染开放的配置项其他参数静默用默认值填充

内置最佳实践 Preset

Preset拓扑自主度默认适合场景
Auto-pilotsequential高（on_risk）批量处理，不需要陪跑
Co-pilothierarchical中（on_uncertainty）日常开发辅助
Manualsequential低（always）学习、敏感操作
Audit-FocusDAG + 回流中代码审计 + 自动修复

Preset 是只读模板，用户选择 Preset 实际上 fork 出一个 snapshot，修改发生在 fork 上，随时可以 diff 或 reset。

7渐进式约束演进路线

约束不是永久限制，而是有数据依据的阶段性护栏。松绑决策基于：成功率数据、失败归因、用户收益估算、降级路径。

约束价值密度

指标 = Δ失败率 / Δ可表达 use case 数。
优先松绑价值密度低的约束（限制多、收益少）。

v1

高护栏期（当前）

  • 允许节点：Planner / Executor / Auditor（三种固定）
  • 允许连接：两条固定边 + 可选回流边
  • 禁止并行分支
  • 暴露参数：仅 autonomy_behavior + tool 开关
  • 目标：成功率 > 90%，用户 0 学习成本
v2

松绑拓扑

  • 开放并行分支（max = 2）
  • 开放自定义节点名称
  • LLM 可生成 DAG 而不只是 sequential
  • 依据：v1 成功率数据 + 失败归因不指向拓扑问题
v3

松绑工具

  • 开放自定义工具接入（用户提供 Tool spec）
  • 开放 ToolRegistry 动态扩展
  • 依据：用户反馈中最高频"希望有但没有"的工具类型

8关键设计权衡

灵活性 vs 成功率

早期约束严格牺牲灵活性，换取高成功率。这是主动选择，不是技术限制。随着数据积累有依据地松绑，不做功能堆砌。

用户控制 vs 系统复杂度

把打断控制权交给用户，意味着系统需要处理"用户不回应"的超时情形。timeout_action 的四种策略覆盖主要场景，复杂度可控。

LLM 生成 vs 用户手动

对话式生成的风险是 LLM 误解意图。缓解：展示 rationale 供用户验证；用 Patch 而非全量重生成；编排在 UI 中完全可见可调。

消息流 vs 传统打断

消息流要求用户主动关注，而非被动被提醒。缓解：支持通知推送；支持消息过滤（只看 Actionable）；流内消息提供快捷操作。
