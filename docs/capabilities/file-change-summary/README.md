# File Change Summary Capability

> Status: planned
> Product Version: Plato 1.0
> Architecture Version: A1
> Owner Area: backend + frontend

## User Problem

Users need to know what files Plato changed, which Task caused each change, and how parent Tasks summarize child changes.

## Current System Capability

- Task UI ViewModels include `TaskFileChangeSummary`.
- Projection service defines a `FileChangeStore` protocol.
- Main Page UX requires File Change Summary and Audit entry.

## Target Capability

Completed TaskNodes show direct file changes. Parent TaskNodes show recursive summaries of all child changes. Session-level views can aggregate the whole change set.

## Known Gaps

| Gap | Plan | Status | Notes |
|---|---|---|---|
| No concrete FileChangeStore | unplanned | open | Need persistent store and query model. |
| No reliable file-change capture pipeline | unplanned | open | Need tool/runtime/workspace integration. |
| Parent-child aggregation not implemented | unplanned | open | Projection has `recursive`; source data missing. |
| No UI detail for change evidence | unplanned | open | Needs file list, summary, and audit link. |
