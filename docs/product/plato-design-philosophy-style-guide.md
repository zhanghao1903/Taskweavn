# 柏拉图 Plato 设计哲学与 UI 风格指南

> Status: UX design baseline
>
> Scope: 产品设计哲学、视觉方向、交互气质、UI 风格约束。本文不是具体页面布局稿，也不是组件 API 文档。

## 1. 核心判断

柏拉图不是 Todo App，不是传统 Task Manager，也不是披着聊天框的 Agent 产品。

柏拉图是一个 Task-first 的智能工作台：

```text
用户表达意图
  -> 系统理解和澄清
  -> 生成可编辑的 TaskTree
  -> 用户确认和补充
  -> 发布 Task
  -> 执行、追踪、验收、审计
```

它的产品价值不是“替用户马上做所有事”，而是把模糊目标转化为可理解、可控制、可追踪的工作结构。

因此，柏拉图的设计哲学应该是：

```text
让复杂工作变得可见。
让用户在执行前理解。
让系统行动保持可控。
让结果具备可追溯的信任感。
```

## 2. 为什么是柏拉图

`柏拉图 / Plato` 这个名字适合这个产品，但原因不应该停留在古典装饰或哲学意象上。

更深层的契合点是：

| 柏拉图意象 | 产品转译 |
|---|---|
| 从现象走向本质 | 从自然语言走向结构化 TaskTree |
| 理性、思辨、秩序 | 清晰的任务、状态、确认和结果 |
| 对复杂世界建模 | WorkTree、TaskNode、Agent routing、Context flow |
| 对真理的追问 | 审计、可追溯、用户信任 |

这个名字应该给产品气质，而不是给 UI 套皮。

我们要避免把柏拉图做成：

- 古希腊装饰主题，
- 大理石雕塑主题，
- 学院派复古页面，
- 神秘主义哲学产品，
- 高概念但低可用的展示品。

更准确的方向是：

```text
现代、克制、理性、可操作的思考工作台。
```

## 3. 品牌人格

柏拉图的品牌人格可以总结为四个词。

### 3.1 清醒

产品应该帮助用户看清事情，而不是制造兴奋感。

表现为：

- 明确的信息层级，
- 清楚的状态表达，
- 稳定的页面结构，
- 可预期的交互反馈，
- 不夸张承诺系统能力。

避免：

- “一键完成一切”的语气，
- 过度拟人化 Agent，
- 让用户误以为系统已经完全理解一切。

### 3.2 有秩序

产品的主角是结构。

表现为：

- TaskTree / WorkTree 是核心视觉对象，
- TaskNode 状态清楚，
- 父子关系、依赖关系、归属关系可见，
- 消息、结果、文件变更都能回到 TaskNode。

避免：

- 所有内容都堆在聊天流里，
- 任务、消息、文件、确认动作互相散落，
- 用户无法判断当前操作影响哪个对象。

### 3.3 克制

克制不是冷淡，而是让用户放心。

表现为：

- 色彩低噪声，
- 动效轻，
- 控件行为稳定，
- 文案直接，
- 重要信息才被强调。

避免：

- 高饱和霓虹 AI 风，
- 大面积炫光和渐变，
- 强玻璃拟态，
- 抢注意力的动画，
- 过度卡片化的营销页面。

### 3.4 可信

柏拉图的惊艳点不是魔法感，而是可理解。

表现为：

- 用户能看到系统如何拆解任务，
- 用户能在风险动作前确认，
- 用户能查看结果和文件变更，
- 用户能进入审计视图追溯过程。

避免：

- 黑箱式自动执行，
- 模糊的“正在思考”长期占位，
- 没有上下文的确认弹窗，
- 完成后只给一句泛泛总结。

## 4. 设计关键词

推荐关键词：

```text
Calm
Structured
Legible
Rational
Trustworthy
Deliberate
```

中文表达：

```text
冷静、清晰、结构化、可理解、可追踪、可控制。
```

不推荐关键词：

```text
Cyber
Neon
Magical
Flashy
Autonomous Employee
One-click Everything
```

柏拉图可以聪明，但不应该炫技。

## 5. 风格方向：Modern Classical Workbench

建议风格方向是：

```text
Modern Classical Workbench
现代理性工作台
```

这里的 Classical 不是装饰性的古典，而是：

- 网格，
- 比例，
- 留白，
- 细线，
- 秩序，
- 平衡，
- 稳定。

这里的 Workbench 是：

- 面向真实任务，
- 信息密度适中，
- 可反复使用，
- 支持扫描、比较、确认和回溯。

因此，柏拉图的 UI 不应该像品牌官网，也不应该像纯聊天工具。它应该像一个经过克制设计的工作界面。

## 6. 视觉原则

### 6.1 布局

布局应服务于“控制面”。

推荐：

- 稳定的多区域工作台布局，
- 左侧或主区域承载 TaskTree / WorkTree，
- 中央承载当前工作流和消息，
- 侧边承载 TaskNode detail / result / file summary，
- 高级审计信息放入单独页面或抽屉。

避免：

- 大面积营销式 hero，
- 卡片套卡片，
- 页面区块全都漂浮成卡片，
- 主操作和审计信息混在一起，
- 用户必须阅读说明文字才知道怎么操作。

### 6.2 信息层级

层级优先级：

1. 当前 Workflow / Session 状态
2. TaskTree / TaskNode 状态
3. 需要用户处理的确认和输入
4. 当前执行进度
5. Result 和 File Change Summary
6. Audit / Logs / Tool details

Main Page 只展示用户下一步需要知道的事实。Audit Page 展示完整证据。

### 6.3 留白

留白用于建立秩序，不用于制造空旷感。

工作台页面需要有足够信息密度。柏拉图不是极简便签，它是复杂任务的控制面。

### 6.4 线条与边界

推荐：

- 细边框，
- 弱分割线，
- 稳定网格，
- 低强度阴影，
- 8px 或更小圆角。

避免：

- 厚重描边，
- 过度阴影，
- 强玻璃质感，
- 大圆角玩具感。

## 7. 色彩系统方向

色彩应表达状态和结构，而不是装饰。

参考视觉稿中的蓝、金、米、灰方向是可用的，但要避免整个产品变成单一蓝色或古典米色主题。

### 7.1 推荐基础色

| Token | 用途 | 建议色值 |
|---|---|---|
| `ink` | 主文本、重要标题 | `#172033` |
| `graphite` | 次级文本、图标 | `#5E6878` |
| `line` | 分割线、边框 | `#DCE3ED` |
| `surface` | 主背景 | `#F7F9FC` |
| `panel` | 面板背景 | `#FFFFFF` |

### 7.2 品牌与操作色

| Token | 用途 | 建议色值 |
|---|---|---|
| `reason-blue` | 主品牌色、主操作 | `#2B5CAA` |
| `clear-blue` | hover、selected、轻强调 | `#5B8FD9` |
| `mist-blue` | 选中背景、轻提示 | `#E8F1FB` |
| `wisdom-gold` | 重要提醒、进行中 | `#D4A137` |
| `classical-cream` | 温和背景、空状态 | `#F5F2EC` |

### 7.3 状态色

| 状态 | 用途 | 建议色值 |
|---|---|---|
| `ready` | 可发布、可执行 | `#4F7D5A` |
| `running` | 正在进行 | `#D4A137` |
| `waiting` | 等待用户 | `#B86E2F` |
| `failed` | 失败、危险 | `#B94A48` |
| `muted` | 已完成、历史项 | `#7A8594` |

### 7.4 色彩使用原则

- 蓝色用于主路径和选中状态。
- 金色用于“注意，但不是危险”。
- 红色只用于真实失败或风险。
- 绿色用于完成和可继续。
- 大面积背景保持低噪声。
- 不用多色渐变表达智能。
- 不用颜色单独表达状态，必须配合文字或图标。

## 8. 字体与排版

### 8.1 字体方向

推荐：

- 中文 UI：`Noto Sans SC` / `Source Han Sans`
- 英文 UI：`Inter`
- 品牌标题或少量强调：可探索 `Noto Serif SC` / `Source Han Serif`

原则：

- UI 正文优先无衬线，保证清晰。
- 衬线体只用于品牌、文档封面、少量仪式感场景。
- 不要在工作台主体里大量使用古典衬线体。

### 8.2 排版原则

- 工作台页面使用紧凑但不拥挤的字号。
- 面板标题克制，不用 hero-scale 字体。
- 按钮文案短。
- 状态文字直接。
- 不使用负字距。
- 不用 viewport width 缩放字体。

## 9. 图标与视觉隐喻

图标应该帮助识别对象，而不是装饰哲学主题。

推荐图标对象：

- Workflow
- Session
- TaskTree
- TaskNode
- Agent
- Message
- Confirmation
- Result
- File Change
- Audit

视觉隐喻可轻微参考：

- 几何结构，
- 节点连接，
- 柱式比例，
- 书页，
- 路径，
- 光束。

但这些隐喻应该抽象化、线性化、低噪声。不要把古希腊柱子、雕塑、神庙作为常驻 UI 元素。

## 10. 交互原则

### 10.1 用户先理解，再授权

对于有后果的动作：

```text
系统说明将要做什么
  -> 用户看到 TaskNode 和影响范围
  -> 用户确认或补充
  -> 系统执行
```

确认动作必须挂在具体对象上，优先挂到 TaskNode。

### 10.2 输入框随上下文改变

同一个自然语言输入，在不同上下文中语义不同：

| 上下文 | 输入语义 |
|---|---|
| 未选中 TaskNode | Session-level 目标、全局补充、新工作流输入 |
| 选中 Draft TaskNode | 对该任务补充、修改、拆分、澄清 |
| 选中 Running TaskNode | 追加指导、提供信息、请求暂停 |
| 选中 Done TaskNode | 追问结果、要求解释、发起 follow-up |

UI 必须让用户感知当前输入会作用于哪里。

### 10.3 TaskTree 是主控制对象

TaskTree 不只是展示，它是用户控制系统的方式。

用户应该能通过 TaskTree 理解：

- 系统计划做什么，
- 哪些任务尚未开始，
- 哪些任务需要确认，
- 哪些任务已经完成，
- 哪些任务失败，
- 哪个 Agent 或能力负责某个 TaskNode。

### 10.4 审计不打扰，但随时可进入

用户在主流程里不应该被原始日志淹没。

但任何重要结果都应该能进入审计视图：

- 查看 Action / Observation，
- 查看确认记录，
- 查看工具调用，
- 查看文件变更，
- 查看风险评估，
- 查看 LLM 请求摘要。

## 11. Motion 与反馈

动效应该表达系统状态，不表达炫技。

推荐：

- 轻微淡入，
- 行内状态变化，
- TaskNode 状态平滑更新，
- 新消息轻微出现，
- 等待状态有稳定节奏。

避免：

- 大面积粒子动画，
- 频繁跳动的 AI thinking 效果，
- 对用户操作造成延迟感的转场，
- 让信息位置难以追踪的动画。

## 12. 文案风格

文案应该像一个冷静、清楚、可靠的协作者。

推荐：

- “需要确认这个文件创建动作。”
- “这个任务将修改 3 个文件。”
- “我需要你补充目标用户。”
- “任务已完成，可以查看结果。”
- “此步骤风险较高，建议先预览。”

避免：

- “我将释放强大的智能代理能力。”
- “让我为你自动搞定一切。”
- “哲思之光正在照耀任务之路。”
- “系统正在深度认知你的意图。”

产品可以有气质，但操作文案必须朴素。

## 13. 组件风格方向

### 13.1 TaskNode

TaskNode 是最重要的 UI 单元。

它应该清楚表达：

- 标题，
- 状态，
- 层级，
- 是否需要确认，
- 是否有结果，
- 是否有文件变更，
- 负责 Agent / Capability。

TaskNode 不应该只是一个普通 Todo 项。

### 13.2 Message

消息应区分：

- 信息，
- 问题，
- 确认请求，
- 用户回答，
- 系统结果，
- 审计摘要。

但视觉上不要分裂成过多类型。优先用轻量标识和行动按钮。

### 13.3 Result Card

Result Card 用于让信息流结果更容易消费。

适合：

- 搜索总结，
- 调研结论，
- 推荐列表，
- 文件变更总结，
- 执行结果摘要。

Result Card 不应该替代 TaskNode，而是 TaskNode 的结果呈现方式之一。

### 13.4 File Change Summary

File Change Summary 应该支持聚合：

- 子 TaskNode 展示直接修改，
- 父 TaskNode 展示所有子节点汇总，
- Session 展示全局汇总。

这是用户验收的重要入口。

## 14. 与参考视觉稿的关系

参考图中的方向有几处值得保留：

- 蓝色作为理性主色，
- 米白和灰色作为低噪声背景，
- 金色作为注意力辅助色，
- 细线和网格感，
- 温和阴影，
- 结构化的控制面截图方向。

需要收敛的地方：

- 雕塑、柱式、学院意象不能成为常驻 UI 主体。
- 古典视觉只能作为品牌气质，不应该压过任务操作。
- UI 信息密度需要更接近真实工作台，而不是品牌展示板。
- “柏拉图”不应该被解释成哲学主题产品，而是结构化思考产品。

## 15. Do / Don't

| Do | Don't |
|---|---|
| 让 TaskTree 成为主对象 | 让聊天流吞掉所有交互 |
| 让确认动作挂到 TaskNode | 弹出无上下文确认框 |
| 用颜色表达状态层级 | 用渐变表达 AI 感 |
| 保留审计入口 | 在主页面堆满日志 |
| 让用户先看懂再执行 | 默认黑箱自动执行 |
| 用古典秩序感指导布局 | 把 UI 做成古希腊主题 |
| 让高级能力逐步展开 | 一开始暴露 Agent / Tool / Bus 全量复杂度 |

## 16. 第一版视觉目标

第一版 Plato UI 的目标不是惊艳截图，而是：

```text
用户第一次打开，就能知道：
1. 我在哪个 Workflow / Session。
2. 系统把我的目标拆成了哪些 Task。
3. 哪些 Task 需要我确认。
4. 哪些 Task 正在执行。
5. 结果在哪里。
6. 如果不放心，在哪里看审计。
```

如果一个不懂技术的用户不用读教程，也能完成这六件事，柏拉图的第一版设计就是成功的。
