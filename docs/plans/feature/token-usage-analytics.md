## 1. Problem

Product 1.0 can execute tasks and already parses provider token usage metadata
when available. However, usage is currently diagnostic/log-adjacent rather than
a product-visible, queryable ledger. Users cannot answer:

- how many tokens did this Task consume?
- how expensive was this Plan?
- how much did this Session consume?
- which Workspace is consuming the most tokens?
- is prompt caching working, and where is it failing?

Product 1.1 needs token usage analytics that are tied to the product model:
Task, Plan, Session, and Workspace.

## 2. Existing Facts

Current implementation already has useful anchors:

- `LLMUsage` includes `input_tokens`, `output_tokens`, `total_tokens`,
  `reasoning_tokens`, `cached_tokens`, `cache_hit_tokens`,
  `cache_miss_tokens`, and `cache_hit_ratio`;
- OpenAI-compatible provider parsing handles DeepSeek/OpenAI-style usage and
  prompt cache fields when present;
- LLM logging records provider/model/request metadata and usage;
- Context Manager cache-aware rendering exposes stable-prefix/cache-friendly
  metadata hooks;
- Product 1.1 multi-workspace routing gives each request a safe workspace
  identity.

Missing layer: durable, queryable, product-scoped usage records and aggregation
APIs.

## 3. Goals

1. Persist one normalized usage event per completed LLM provider call.
2. Attribute usage to Workspace, Session, Plan, Task, Agent run, provider,
   model, and request purpose when those dimensions are available.
3. Aggregate usage by Task, Plan, Session, and Workspace.
4. Show cache effectiveness with explicit provider-reported cache metrics.
5. Keep raw prompts, completions, provider payloads, secrets, logs, and SQLite
   rows out of product analytics APIs.
6. Support unknown/partial provider usage without fabricating precision.
7. Make the first backend slice useful without requiring billing-price support.

## 4. Non-Goals

- No dollar-cost billing calculation in the first slice.
- No quota enforcement or budget stop policy.
- No external analytics service.
- No prompt/content inspection UI.
- No per-user/team cloud accounting.
- No automatic optimization recommendations.
- No provider network validation beyond existing LLM calls.

## 5. Metric Definitions

### 5.1 Token Counters

| Metric | Meaning |
|---|---|
| `inputTokens` | Tokens sent to the provider, if reported. |
| `outputTokens` | Tokens generated by the provider, if reported. |
| `reasoningTokens` | Reasoning/thinking tokens, if reported separately. |
| `totalTokens` | Provider total if reported; otherwise sum of known input/output/reasoning when safe. |
| `cachedTokens` | Provider-reported cached prompt tokens. |
| `cacheHitTokens` | Provider-reported prompt cache hit tokens. |
| `cacheMissTokens` | Provider-reported prompt cache miss tokens. |
| `callCount` | Count of LLM calls included in the aggregate. |
| `unknownUsageCallCount` | Calls where the provider did not return token usage. |

### 5.2 Cache Hit Rate

First Product 1.1 definition:

```text
providerCacheHitRate = sum(cacheHitTokens) /
  (sum(cacheHitTokens) + sum(cacheMissTokens))
```

Fallback when `cacheMissTokens` is unavailable but `inputTokens` and
`cacheHitTokens` are available:

```text
providerCacheHitRate = sum(cacheHitTokens) / sum(inputTokens)
```

Rules:

- do not compute a rate if the denominator is missing or zero;
- label the rate source as `hit_miss_tokens`, `input_tokens`, or `unavailable`;
- keep local Context Manager cache-friendly prefix metadata separate from
  provider-reported cache hits.

## 6. Attribution Model

Every usage record should include best-effort dimensions:

| Dimension | Required? | Notes |
|---|---:|---|
| `workspaceId` | Yes | Use safe workspace id, never raw absolute path. |
| `sessionId` | Yes when call is session-bound | Authoring and execution should be session-bound. |
| `planId` | Optional initially | Product 1.1 Plan model may be absent or evolving. |
| `taskId` / `taskNodeId` | Optional | Present for execution calls; null for workspace/session setup calls. |
| `agentRunId` | Optional | Useful for Audit and diagnostics correlation. |
| `requestPurpose` | Yes | Examples: `authoring`, `task_execution`, `context_checkpoint`, `settings_check`. |
| `provider` | Yes when known | Redacted provider metadata only. |
| `model` | Yes when known | Safe model id, no API key or routing payload. |

If a call cannot be attributed to a Task, it must still be counted at Session
or Workspace level when possible. Unattributed task rows must not be guessed.

## 7. Data Model

Recommended durable store:

```text
.plato/usage.sqlite
```

First table:

```text
llm_usage_events
  usage_event_id text primary key
  occurred_at text not null
  workspace_id text not null
  session_id text null
  plan_id text null
  task_node_id text null
  agent_run_id text null
  request_purpose text not null
  provider text null
  model text null
  provider_request_id_hash text null
  input_tokens integer null
  output_tokens integer null
  total_tokens integer null
  reasoning_tokens integer null
  cached_tokens integer null
  cache_hit_tokens integer null
  cache_miss_tokens integer null
  cache_hit_ratio real null
  usage_source text not null
  cache_rate_source text not null
  metadata_json text not null
```

The store must not persist raw prompts, completions, tool arguments, provider
payloads, secrets, or absolute workspace paths.

## 8. API Slices

### U1. Backend Usage Ledger

Status: implemented.

- Add normalized usage event model and SQLite store.
- Record usage from LLM response metadata at the provider/client or AgentLoop
  boundary where product dimensions are known.
- Add focused tests for redaction, null usage, and cache metric normalization.

### U2. Query API

Status: implemented.

- Add workspace-scoped usage summary endpoint.
- Support dimensions: `task`, `plan`, `session`, `workspace`.
- Support time window filters and optional session/task filters.
- Return aggregate rows plus totals.

### U3. Main Page / Task Detail Read Model

Status: implemented.

- Add Task and Plan usage summary to Main Page detail panels.
- Keep it compact: token total, input/output split, cache hit rate.
- Do not show provider raw metadata in Main Page.

### U4. Workspace Usage View

Status: implemented as a simple Product 1.1 surface.

- Add a simple Workspace-level usage surface that can compare Sessions and
  Plans.
- Show unknown usage call count so users understand incomplete provider data.

### U5. Audit / Diagnostics Links

Status: diagnostics implemented; Audit event-id linkage deferred.

- Let Audit records reference usage event ids or aggregate ids.
- Include redacted usage summaries in diagnostic bundles.

### U6. Cache Analytics Hardening

Status: deferred follow-up.

- Correlate provider cache hit metrics with Context Manager render mode and
  stable prefix metadata.
- Keep this as analysis metadata, not a user-facing promise that cache will hit.

## 9. Frontend Requirements

Initial UI should expose:

- Task detail: total tokens and cache hit rate for that Task;
- Plan overview: aggregate total tokens and cache hit rate across included
  Tasks plus authoring calls attributed to the Plan when available;
- Session summary: total tokens, call count, unknown usage call count;
- Workspace usage view: totals by Session and recent Plans.

Display rules:

- show `Not reported` when provider usage is unavailable;
- show `Cache unavailable` when cache denominator is unavailable;
- show percentages with one decimal place only when calculated from reported
  provider data;
- do not imply dollar cost until price tables and model mapping are accepted.

## 10. Acceptance Criteria

- Token usage events are stored without raw prompt/completion/provider payloads.
- Queries can aggregate by Task, Plan, Session, and Workspace.
- Cache hit rate is only calculated from documented provider metrics.
- Unknown provider usage is visible and does not silently drop calls.
- Diagnostics include redacted usage summaries.
- Focused tests cover usage normalization, aggregation, cache rate fallback,
  and workspace path redaction.

## 11. Decisions And Deferred Items

1. `planId` uses explicit metadata when available; task execution falls back to
   the current TaskBus root id until first-class Plan identity lands.
2. Authoring calls without a published/known Plan remain Session/Workspace
   usage instead of being guessed into a Plan candidate.
3. Cache-friendly Context Manager prefix metadata stays out of the first
   user-facing UI; provider-reported cache hit rate is shown with source labels.
4. Model price tables, budget enforcement, and quota stop policy are deferred
   until a separate pricing/budget contract is accepted.

## 12. Implemented Slice

Implemented Product 1.1 slice:

- durable `llm_usage_events` store;
- capture normalized usage from real provider calls;
- workspace/session/task best-effort attribution;
- workspace-scoped query endpoint;
- contract tests for Task/Plan/Session/Workspace aggregation and cache hit rate;
- Main Page compact Task/Plan usage cards;
- Workspace Usage route with Workspace/Session/Plan/Task breakdown sections;
- diagnostic bundle redacted usage summary.
