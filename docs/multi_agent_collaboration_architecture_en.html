Multi-Agent Collaboration Architecture

v1.0 · 2026-05-07 · codeAgent Phase 3–4  |  中文版

1Core Design Principles

The entire architecture is built around two orthogonal principles that together govern how agents and users collaborate.

Principle 1

Stream over Interrupt

Traditional blocking interrupt models have the system drive the pace, forcing users to respond. The new model replaces this with a message stream: agents post requests to the stream; users respond at will. The user is always the active party.

Principle 2

Constraint-driven Orchestration

Users are not forced to understand graph structures. The LLM acts as orchestration designer, generating valid agent graphs within constraint boundaries. Users describe intent in natural language and make local adjustments via the UI. Constraints evolve as the system matures.

The Core Shift

Old model: interrupt → pause → resume    (blocking, system-driven)
New model: agent posts message → user may or may not respond  (stream, user-driven)

Old model: user fills in orchestration config  (fill-in-the-blank)
New model: describe intent → LLM generates draft  (multiple choice)

Old model: constraints are post-hoc validators  (generated then rejected)
New model: constraints are generation-time context  (valid by construction)

2Message Stream Model

All user-facing communication produced during agent execution flows into a single Message Stream. Users can observe in real time and intervene at any point, but are never forced to block.

Stream Example

INFO Planner: execution plan generated — 5 steps, 3 files affected
ASK Executor: about to delete config.py — this is irreversible, confirm?
Waiting 120s; timeout action: skip this step
INFO Executor: auth.py lines 42–58 modified, change recorded
WARN Auditor: potential SQL injection vulnerability detected (confidence 0.87) — recommend manual review

Two Message Types

TypeMeaningAgent BehaviourUser Behaviour
InformationalTells user what the agent didSends and continues immediatelyMay ignore; can review later
ActionableRequests user input or confirmationWaits per autonomy configRespond or ignore; timeout behaviour configurable
"Never Interrupted" State

When autonomy is at maximum, every Actionable message has a timeout with automatic proceed, and the stream degrades to a read-only execution log. The user can inspect it at any time, but nothing blocks execution. This is the user's active choice — the system's only job is to surface the quality/autonomy trade-off clearly.

3Autonomy System

Autonomy is not a magic number — it is a precise description of "how the agent acts under uncertainty." Two orthogonal dimensions: trigger (when to send an Actionable message) and wait (how long to wait, what to do on timeout).

AutonomyBehavior Configuration

@dataclass
class AutonomyBehavior:
    # Trigger dimension: when to send an Actionable message
    trigger: Literal[
        "never",             # never; all messages are Informational
        "on_risk",           # high-risk operations only (delete, exec, etc.)
        "on_uncertainty",    # when LLM confidence falls below threshold
        "always",            # every key action requests confirmation
    ]
    confidence_threshold: float   # 0.0–1.0; on_uncertainty only

    # Wait dimension: behaviour after an Actionable message is sent
    wait_timeout: float | None    # seconds; None = wait indefinitely
    timeout_action: Literal[
        "wait",              # keep waiting (low-autonomy default)
        "proceed_default",   # continue with most conservative choice
        "proceed_confident", # continue with highest-confidence LLM choice
        "skip",              # skip the action entirely
    ]
    notify_on_proceed: bool       # notify user when agent self-decides on timeout

Preset Autonomy Levels

Full autonever——Batch jobs, reversible tasks

Quality vs Autonomy Trade-off

High Autonomy

  • Uninterrupted execution, fast
  • Low cognitive load for user
  • Agent guesses on ambiguous situations
  • Errors may accumulate silently

Low Autonomy

  • User participates in every key decision
  • Low error rate, high controllability
  • Requires continuous user attention
  • Speed depends on user response time

The choice belongs entirely to the user. The system makes no judgement — it only surfaces the trade-off clearly in the UI.

4Conversation-driven Orchestration

Users do not need to understand graph structures. The Orchestration Designer (a meta-agent) translates natural-language intent into a valid agent graph within constraint boundaries. Users only select and fine-tune.

User natural language intent
        ↓
OrchestrationDesigner (meta-agent)
  Input:  user intent + ConstraintProfile + ToolRegistry
  Output: OrchestrationDraft (valid graph + node configs + rationale)
        ↓
UI renders graph  ←→  user fine-tunes (multiple choice, not free text)
Key Principle

Constraints are generation-time context, not post-hoc validators. The LLM generates within constraint bounds; the output is valid by construction — eliminating "UI allows but backend rejects" inconsistencies.

Three Generation Phases

# Phase 1: Intent Parsing
User: "I want a system that audits code for security vulnerabilities and auto-fixes them"
→ Core capabilities: [code reading, vuln analysis, fix execution, validation loop]
→ Risk identified:   fix execution = high-risk, requires user confirmation
→ Constraint map:    auditor node required; interrupt before fix step

# Phase 2: Constraint-aware Graph Generation
LLM receives ConstraintProfile as prompt context
Generates graph within allowed topology — no backend validator needed

# Phase 3: Capability Assignment
Each agent node selects tools from ToolRegistry
Tools are selected, not generated  ← system security floor

Conversation as Diff, Not Full Regeneration

User feedback on the draft triggers a local patch, not a full regeneration. The UI highlights the changed node; the user sees a diff, not an entirely new graph to re-understand.

User: "make the executor more autonomous"

DraftPatch:
  target_node: "executor"
  changes:
    autonomy_behavior.trigger: "on_uncertainty" → "on_risk"
    autonomy_behavior.confidence_threshold: 0.7 → 0.4
  reason: "Reduce interruption; executor tries first rather than asking"

OrchestrationDraft Structure

@dataclass
class OrchestrationDraft:
    nodes: list[AgentNodeDraft]
    edges: list[EdgeDraft]
    rationale: str                       # LLM explains the design; shown to user

@dataclass
class AgentNodeDraft:
    id: str
    agent_type: AgentType
    display_name: str
    description: str                     # this node's role in the orchestration
    tool_set: list[ToolRef]              # subset selected from ToolRegistry
    autonomy_behavior: AutonomyBehavior
    suggested_alternatives: list[AgentType]  # types the user may swap this node to

suggested_alternatives turns node replacement into a multiple-choice question — users don't need to know what agent types exist.

5Constraint-driven UI Orchestration

Layered Configuration Model

Layer 3: Custom DAG (advanced users)
  Freely connect agent nodes within constraint bounds, adjust any parameter

Layer 2: Preset + parameter tuning (intermediate users)
  Pick a best-practice template, adjust key parameters via sliders

Layer 1: Preset selection (regular users)
  Auto-pilot / Co-pilot / Manual / Audit-Focus

All three layers normalise to the same OrchestrationConfig internally.

ConstraintProfile: Constraints as First-class Citizens

@dataclass
class ConstraintProfile:
    version: str

    # Node dimension: which agent types the user may place
    allowed_agent_types: set[AgentType]

    # Edge dimension: which connections are allowed
    allowed_edges: list[EdgeRule]          # (src_type, dst_type, condition)
    forbidden_patterns: list[Pattern]      # e.g. no executor→executor direct link

    # Composition dimension
    required_nodes: set[AgentType]         # e.g. auditor must always be present
    max_parallel_branches: int

    # Meta: why each constraint exists
    rationale: dict[str, str]              # constraint_key → failure mode description
rationale is not a comment

rationale is the input to loosening decisions — it tells you what failure mode you're betting against when you remove a constraint. With it, "should we loosen this?" becomes a data-backed engineering decision, not a product hunch.

7Progressive Constraint Evolution

Constraints are not permanent restrictions — they are staged guardrails backed by data.

Constraint Value Density

Metric = Δfailure rate / Δexpressible use cases.
Prioritise loosening low-density constraints — those that restrict a lot but unlock little.

v1

High Guardrails (current)

  • Allowed nodes: Planner / Executor / Auditor (three fixed types)
  • Allowed edges: two fixed edges + optional feedback loop
  • Parallel branches: forbidden
  • Exposed params: autonomy_behavior + tool toggles only
  • Goal: success rate > 90%, zero user learning curve
v2

Loosen Topology

  • Open parallel branches (max = 2)
  • Allow custom node display names
  • LLM can generate DAGs, not just sequential flows
  • Basis: v1 success rate data + failures not caused by topology
v3

Loosen Tools

  • Allow custom tool integration (user provides Tool spec)
  • Open ToolRegistry to dynamic extension
  • Basis: most-requested "wish we had" tool types from user feedback

8Key Design Trade-offs

Flexibility vs Success Rate

Strict early constraints sacrifice flexibility for high success rates. An intentional choice, not a technical limitation. Loosening happens with data backing, not feature accumulation.

User Control vs System Complexity

Giving users interruption control means handling all "no response" timeout scenarios. The four timeout_action strategies cover the main cases at manageable complexity.

LLM Generation vs Manual Config

Risk: LLM misreads intent. Mitigations: show rationale for user verification; use Patch not full regeneration; orchestration fully visible and editable in UI at all times.

Message Stream vs Traditional Interrupt

Cost: users must actively monitor stream. Mitigations: push notifications for Actionable messages; stream filtering (Actionable only); quick-action buttons inline in messages.
